# 블로그 품질 기준

> **이 파일은 사용자가 직접 진화시킵니다.** Claude는 명시적 요청 없이 이 파일을 수정하지 마세요.
>
> 이 기준은 `blog_agent_pipeline.py`의 QA 에이전트(Sonnet 4.6)가 채점할 때 시스템 프롬프트로 사용됩니다.

## 합격 기준 (모두 충족해야 통과)

### 1. 구조

- 도입 (2-4문장): 독자가 왜 끝까지 읽어야 하는지가 첫 단락에 있다
- 본문: 최소 1개의 구체적 사례 또는 데이터
- 결론·CTA: 독자가 다음에 무엇을 할지 명확

### 2. 분량

- 한국어 기준 **1,500–2,500자** (공백 제외).
- 너무 짧으면 깊이 부족, 너무 길면 이탈.

### 3. 톤

- 1인칭 사업자 시점. "저는", "제가". (3인칭 분석가 시점 금지)
- 단정문 위주. "~할 수도 있습니다", "~일 것입니다" 같은 약한 표현 최소화.
- 추상적 마케팅 용어 ("혁신적", "최고의", "유일한") 금지.

### 4. 신뢰성

- 수치·인용·사례는 출처가 명시되거나 본인 경험임이 명확.
- 출처 없는 수치 단정 금지 ("매출이 3배 증가했습니다" without context → 실패).

### 5. 가독성

- 한 단락 5문장 이하.
- 헤더(##, ###) 2-4개로 본문 구조화.
- 불릿 리스트는 3-7개 사이.

## 즉시 실패 (하나라도 해당 시 점수 무관 실패)

- 사실관계 오류 (검증 가능한 거짓 정보)
- 표절 의심 (소스 PDF와 문장 단위 일치)
- 사용자의 1인칭 톤이 아닌 일반 블로그체로 회귀
- 제목과 본문 불일치
- 결론이 도입부 약속을 배신

## QA 출력 포맷

QA 에이전트는 다음 JSON으로 응답합니다:

```json
{
  "pass": true,
  "score": 85,
  "rubric": {
    "structure": 9,
    "length": 10,
    "tone": 8,
    "credibility": 8,
    "readability": 9
  },
  "instant_fails": [],
  "improvements": [
    "도입부 약속을 결론에서 다시 한 번 환기하면 더 강함"
  ],
  "verdict": "통과. 발행 가능."
}
```

- `pass: true` → `blog-queue/`에 status=draft로 저장
- `pass: false` → 1회 재작성 시도, 그래도 실패하면 사람 검토용으로 저장
